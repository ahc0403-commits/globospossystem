# Flare Memory

- For POS integration plans, not-yet-implemented Deliberry operational order flows must be classified as Blocked or Decision Needed until implementation and contracts are confirmed. Reserve No-Go for unsafe failures, data contamination, broken coupling, or attempts to claim Full Go while D1 remains blocked.
- Deliberry operational order state machines must never let `DELIBERRY_ORDER_RECEIVED`/`new` downgrade an actioned or terminal order. Stale or out-of-order events belong in anomaly/reconciliation evidence, not state mutation.
- Do not claim POS app pilot execution PASS or 70% user-test PASS from account auth/profile checks alone. A successful app-level user-flow smoke or equivalent command-backed evidence for at least 35/49 pilot cases is required.
- Do not claim POS production deploy readiness unless required POS pilot Auth accounts are checked against production Supabase Auth/profile data, or the Auth check is explicitly skipped and reported as a blocker-risk. Deployment automation must not create, print, or reset pilot passwords.
- POS pilot Auth/profile readiness checks must mirror the app login path: public.users is resolved by public.users.auth_id = auth.users.id. Do not join public.users.id to auth.users.id when judging login readiness.
- Print-routing enqueue batch numbers must be computed once per enqueue event before looping copy types; never recompute after inserting the first destination, or kitchen/floor delta jobs drift into different batch numbers.
- SECURITY DEFINER database functions that set a restricted search_path must schema-qualify extension functions, and contract tests should exercise the callable RPC path rather than only inspecting function text.
- SQL contract tests that depend on auth.uid()-based roles must seed the required auth/users fixture rows inside the transaction instead of relying on local or production seed data.
- POS migrations must be replay-safe: guard table and policy creation against existing objects.
- Legacy `SECURITY DEFINER` RPCs must revoke execution from `PUBLIC`, `anon`, and `authenticated` unless they enforce internal `owner_type` and validate the complete entity-brand-store tuple while preserving required internal compatibility.
- Production migrations must preflight every referenced optional schema, table, and helper; missing dependencies must block before mutation or be explicitly deferred.
- Reassignment operations must preserve immutable ownership and assignment history, including actor, timestamp, source, and destination.
- Production deployment scripts must pin approved project and environment targets; runtime target overrides are forbidden.
- Every production migration must include executable preflight, verification, and rollback assertions for prerequisites, invariants, and restoration behavior. Completion requires a real non-bypassed production connection from clean approved exact-main source, a read-only preflight, controlled apply and verification, and sanitized recorded evidence; dry runs or blank fields prove neither connectivity nor deployment.
- Migration backups must remain immutable across replays, and rollback tests must prove exact original-state restoration with all migration-generated history absent.
- Verification SQL must assert every required backup object and use anti-join negative tests so backed-up records missing from the live table cannot be hidden by inner joins.
- Mutation RPCs that create or change legal-entity assignments must atomically maintain append-only active-period history and include executable RPC tests.
- Run production SQL with fail-fast single-transaction semantics (`psql -X -v ON_ERROR_STOP=1 --single-transaction --file`) and emit success only after exit code 0; validate the exact deployment invocation rather than substituting `supabase db query --linked -f` for multi-statement scripts.
- Collector preflight must validate every first-write table and permission, not only aggregate or read paths.
- Parse collector configuration strictly: reject unknown boolean values and misspelled keys instead of silently disabling a store.
- Collector failure reporting must run under an unconditional cleanup path so setup and browser failures cannot bypass escalation.
- Persist and audit an explicit scheduled-slot identity; never infer a delayed run's slot from its actual start timestamp.
- Persist the intended local schedule date and time independently of execution time, with a cross-midnight delayed-run test preserving the intended final 22:30 attribution.
- Photo Objet scheduled collection uses non-overlapping HCM intervals ending at 10:00, 12:00, 14:00, 16:00, 18:00, 20:00, and 22:30; the final interval is 20:00-22:30, and health must allow at least 90 minutes for GitHub scheduler delay before marking a slot missing.
- Never continue a migration after a history mismatch or unresolved confirmation warning; stop and verify the applied migration chain first.
- Block production deploys from dirty worktrees or commits not descended from the approved main branch.
- 번역 리소스가 존재하는 것만으로 다국어 완료로 판단하지 않는다. 사용자에게 보이는 정적 문구·식별자 레이블과 백엔드·도메인 열거형 값, 경고, 오류, 상태, 코드는 지원 언어별 완전한 매핑으로 현지화하고, 계약 테스트로 현지화 API 사용과 원시 레이블·값·코드 미노출을 검증한다.
- 운영 배포 스크립트는 캐시나 기존 `.dart_tool`에 의존하지 않고, 클린 작업 트리에서 필요한 의존성을 먼저 설치한 뒤 검증해야 한다.
- Supabase CLI 임시 `cli_login` 자격 증명으로 운영 SQL을 실행할 때는 고정된 `postgres` 역할 전환을 명시하고, mutation 전에 실제 `current_user`를 검증해야 한다.
- 배포 라이브러리를 source하는 셸 wrapper는 라이브러리 소유 readonly 변수를 미리 선언하지 않으며, wrapper 진입점 자체를 실행하는 테스트로 고정 대상과 fail-fast 경로 도달을 검증한다.
- Supabase default ACL may grant `service_role` broad table rights; migrations that narrow a control-plane table must explicitly `REVOKE ALL ... FROM service_role` before granting read-only access, and tests must reproduce a broad default ACL.
- Health state and alert delivery state are independent: acknowledging an alert may suppress duplicate delivery but must never make a persistent unhealthy condition pass.
- Expected-work monitoring must compare every effective policy with its full materialized expectation set; checking only rows that already exist fails open when one policy or slot was never generated.
- A production control-plane migration that requires policy rows must apply schema, approved policy configuration, and initial materialization in one fail-fast transaction before verification.
- Exact-interval zero-sales recovery must use an interval-specific row count or flag, never a daily aggregate row count.
- Bash 3.2 with `set -u` rejects expansion of an empty array; production runners must branch before array expansion and the frozen validation gate must inspect each command's exit code, not infer success from empty output.
- Health lookback may bound healthy history but must always union every unresolved `missing` or `failed` slot so persistent operational debt cannot age into PASS.
- Collection preflight may fail closed on sales-write targets and store identity, but monitoring-ledger probes are best-effort observability and must never block an otherwise valid collection.
- Migration state tables that hold rollback definitions are owner-only control-plane objects; revoke all service-role access and test them under broad default ACLs before using their contents in dynamic rollback SQL.
- Rollback evidence tables must have deterministic row counts and fingerprints anchored in owner-only state; verification and rollback must authenticate every evidence table before mutation, with deleted and modified evidence tests.
- A release-proof state label is not evidence: CODE_VALIDATED must be backed by a successful full validation run whose head SHA exactly equals the current origin/main SHA, and absent, failed, PR-only, or stale runs must fail closed.
- A main-only operational proof with workflow_dispatch must reject every non-main ref before checking out or executing repository code; validating a main SHA later does not make an untrusted initial ref safe.
- Replay-safe rollback must restore pre-existing column constraints and comments exactly, not merely preserve the column; tests need a fixture where the target column already exists with different metadata.
- Release evidence must persist the production provider's confirmed deployment SHA, not only validate it transiently; the artifact must prove main_sha, validated_sha, and deployment_sha are identical after source metadata is discarded.
- A workflow_run proof must compare the event SHA with the live main ref through trusted provider metadata before its first checkout; a post-checkout origin/main comparison is still required for race detection but cannot replace the pre-checkout stale-run guard.
- Local Worker and Judge success is only preflight evidence; a release gate is not PASS until required GitHub Linux checks succeed on the exact pushed head SHA.
- Cross-platform validation runners must explicitly propagate every invoked process's nonzero exit code, and text assertions must normalize platform line endings. Tests must construct required Git fixtures explicitly and never rely on shell-version-specific error behavior to reach an assertion.
