# GLOBOS POS Toast 운영 원칙 기반 화면 재구성 제안

## 1. 목적

이 문서는 GLOBOS POS의 현재 기능 구조를 유지하면서도, Toast POS의 운영 원칙인 `Queue -> Select -> Act -> Optional Detail`을 기준으로 화면 책임과 역할별 흐름을 더 명확히 재구성하기 위한 제안서다.

핵심 방향은 아래와 같다.

- 주문, 주방, 결제, 관리 업무를 각각 더 선명한 운영 큐로 재구성
- 한 화면에 너무 많은 기능을 동시에 노출하지 않고, 주요 액션 중심으로 단순화
- 베트남 로컬 요구사항과 본사 운영 기능은 유지하되, 메인 플로우와 예외 플로우를 분리
- Toast의 장점인 단순하고 직관적인 작업 순서를 참고하되, GLOBOS POS의 차별 기능은 독립 워크스페이스로 강화

## 2. 재설계 원칙

### 2.1 공통 운영 모델

모든 운영 화면은 아래 순서를 기본으로 삼는다.

1. Queue
2. Select
3. Act
4. Optional Detail

의미는 다음과 같다.

- Queue: 지금 처리해야 할 항목이 먼저 보여야 한다.
- Select: 사용자가 현재 무엇을 선택했는지가 명확해야 한다.
- Act: 다음에 해야 할 안전한 행동이 가장 눈에 띄어야 한다.
- Optional Detail: 세부 정보는 보조적이어야 하며 메인 작업 흐름을 가리지 않아야 한다.

### 2.2 현장형 UI/UX 원칙

레스토랑 운영 환경은 빠르고, 반복적이며, 실수 비용이 높다. 따라서 아래 원칙은 개별 화면 개선보다 상위에 있는 공통 설계 기준으로 취급해야 한다.

#### 단순화 유지

- 메인 액션 외의 요소는 과감히 숨긴다.
- 한 화면에서 동시에 이해해야 할 개념 수를 줄인다.
- 드물게 쓰는 기능은 오버플로, 모달, 드로어로 보낸다.

#### 업무 순서에 따른 구성

- 화면은 시스템 구조가 아니라 직원의 실제 절차 순서대로 배치한다.
- 한 단계에서 필요한 정보와 변수만 보여 준다.
- 다음 단계로 넘어가기 전까지 불필요한 판단 요소는 숨긴다.

#### 일관성 강조

- 버튼 위치, 색상 의미, 주요 아이콘, 리스트/상세/오버플로 패턴을 전 화면에서 반복한다.
- 같은 성격의 행동은 항상 같은 자리와 같은 시각 언어를 사용한다.
- 관리자 탭 구조도 공통 패턴으로 통일한다.

#### 색상·형태 코딩

- 상태와 카테고리는 학습 가능한 규칙으로 고정한다.
- 예를 들어 `진행 중`, `완료`, `주의`, `예외`를 각기 다른 색상과 배지 형태로 통일한다.
- 테이블 카드, 주문 카드, 정산 카드 모두 같은 상태 언어를 공유하도록 한다.

#### 역할 기반 워크플로우

- 각 역할은 자신의 업무에 필요한 정보만 기본적으로 본다.
- 관리자, 웨이터, 캐셔, 주방, 본사 운영자가 보는 기본 화면은 서로 달라야 한다.
- 권한은 단순 접근 제한이 아니라 "보이는 정보의 범위와 깊이"까지 포함해 설계한다.

### 2.3 화면 구조 원칙

- 리스트, 큐, 컬럼, 테이블, 칸반 중심으로 구성한다.
- KPI나 상세 정보는 메인 큐 아래 또는 옆으로 내린다.
- 드물게 쓰는 기능은 모달, 드로어, 오버플로 메뉴로 보낸다.
- 운영자가 즉시 수행해야 할 액션은 큰 버튼과 명확한 우선순위로 노출한다.
- 지역 특화 기능은 숨기지 말고, 메인 플로우와 예외 큐를 분리해 가시성을 높인다.

## 3. 화면별 재구성 제안

## 3.1 웨이터 화면

### 현재 문제

- 테이블, 주문, 추가 작업이 한 흐름으로 섞여 보일 수 있다.
- 테이블 이동, 추가 주문, 게스트 수 같은 부가 기능이 메인 주문 흐름을 방해할 여지가 있다.

### 목표 플로우

1. 테이블 선택
2. 주문 추가 또는 기존 주문 이어받기
3. 주문 검토
4. 주문 전송

### 제안 구조

- 첫 화면은 `테이블 큐`에 집중한다.
- 테이블 카드는 아래 정보를 한 번에 보여 준다.
  - 테이블 번호
  - 게스트 수
  - 현재 주문 금액
  - 경과 시간
  - 담당 서버
  - 상태 색상
- 테이블 선택 후 주문 편집 화면으로 들어간다.
- 주문 편집 화면은 3분할로 구성한다.
  - 좌측: 체크 요약
  - 중앙: 메뉴 탐색
  - 우측 또는 하단 고정: `전송`, `취소`, `결제 요청`
- 버페/하이브리드의 게스트 수 입력은 모달로 처리한다.
- 테이블 이동, 주문 취소, 서버 변경은 오버플로 메뉴로 내린다.

### 기대 효과

- 웨이터는 "어느 테이블을 처리해야 하는가"와 "지금 무엇을 전송해야 하는가"에만 집중할 수 있다.
- 드문 기능이 메인 주문 작업을 방해하지 않는다.
- 주문 화면이 절차 순서대로 읽히기 때문에 신규 직원 교육 비용도 줄어든다.

## 3.2 주방 화면

### 현재 문제

- 주문 카드 중심 구조는 적절하지만, 스테이션 분리나 예외 시나리오가 커질수록 복잡성이 증가할 수 있다.

### 목표 플로우

1. 주문 큐 확인
2. 주문 또는 아이템 선택
3. 상태 변경

### 제안 구조

- 화면 중심은 항상 `주방 큐`다.
- 주문은 상태별 컬럼 또는 명확한 리스트 구분으로 배치한다.
  - Pending
  - Preparing
  - Ready
- 오래된 주문은 시간 기반 강조 색상 또는 배지로 표시한다.
- 상단 액션은 최소화한다.
  - All Day View
  - Recall
  - Settings
- 라인 분화가 필요해지면 모드를 나눈다.
  - Expediter Mode: 전체 티켓 확인
  - Prep Station Mode: 특정 조리 라인 전용 큐

### 기대 효과

- 주방은 정보 탐색보다 상태 처리에 집중할 수 있다.
- 향후 스테이션 분화가 필요할 때도 구조 확장이 쉽다.
- 색상과 위치가 일관되면 주방 직원은 상태를 텍스트보다 빠르게 인식할 수 있다.

## 3.3 캐셔 화면 및 결제 플로우

### 현재 문제

- 결제 이후 증빙, 전자세금계산서, 영수증 등의 후속 흐름이 한 덩어리로 느껴질 수 있다.
- 베트남 로컬 요구사항이 강한 만큼, 결제 직후 예외 처리 책임을 어디까지 보여 줄지 더 분명해야 한다.

### 목표 플로우

1. 결제 대상 선택
2. 결제 수단 선택
3. 결제 처리
4. 증빙/전자세금계산서 후속 안내
5. 영수증 마무리

### 제안 구조

- 좌측 패널에는 항상 결제 대상 주문과 금액 요약을 고정한다.
- 우측 패널에는 큰 액션 버튼을 둔다.
  - 카드
  - 현금
  - 기타 결제
  - 분할 결제
  - 리워즈
  - 서비스 차지
- 팁 선택은 별도 팝업 또는 별도 단계로 분리한다.
- 결제 완료 후 후속 액션은 순차적으로 보여 준다.
  - 영수증 전달
  - 증빙 사진 업로드
  - E-Invoice 접수 상태
- 전자세금계산서 발행은 결제 직후 백그라운드 큐로 보낸다.
- 실패/지연/재시도는 캐셔 메인 화면이 아니라 별도 `E-Invoice 현황` 워크스페이스에서 관리한다.

### 기대 효과

- 캐셔는 결제라는 본업에 집중할 수 있다.
- 전자세금계산서의 중요성은 유지하면서도, 결제 플로우 자체는 단순해진다.
- 후속 예외 처리가 메인 결제 단계와 분리되어 결제 속도와 정확도가 모두 좋아진다.

## 3.4 관리자 탭 구조

### 현재 문제

- `admin` 탭 범위가 매우 넓다.
- 한 탭 안에서도 작업, 설정, 이력, 예외 처리가 함께 섞일 수 있다.

### 목표

- 각 탭은 "지금 처리할 일"과 "선택한 항목의 상세/설정"을 분리해야 한다.
- 검색, 필터, 리스트 중심으로 재구성해야 한다.

### 공통 제안

- 각 탭의 첫 화면은 `작업 큐 또는 리스트`가 되어야 한다.
- 세부 설정, 히스토리, 고급 옵션은 드로어 또는 상세 패널로 이동한다.
- 상단은 필터와 핵심 액션만 유지한다.
- 각 탭은 동일한 탐색 규칙을 가져야 한다.
  - 좌측 또는 상단: 리스트/필터
  - 중앙: 선택 항목
  - 우측 또는 하단: 상세/보조 액션

### 탭별 제안

#### Tables

- 좌측: 테이블 목록 또는 레이아웃 목록
- 우측: 선택 테이블 상세/편집
- 추가/삭제/배치 변경은 우측 상세 또는 툴바에서 처리

#### Menu

- 좌측: 카테고리 트리
- 중앙: 메뉴 리스트
- 우측: 선택 메뉴 상세/가격/연결 레시피

#### Staff

- 리스트와 검색을 우선
- 직원 선택 후 권한/역할/매장 접근 범위를 우측 상세에서 편집

#### Reports

- KPI보다 `운영 주의 큐`를 먼저 배치
- 이후 일별/유형별 수치와 리포트 export 제공
- 마감은 리포트의 부속 기능이 아니라 별도 명확한 액션 블록으로 배치

#### Inventory

- 재고 화면은 단계형 흐름으로 정리
  1. 원재료
  2. 레시피
  3. 발주 추천
  4. 승인
  5. 입고 확인
- 좌측 리스트, 우측 상세 패턴을 사용
- 히스토리와 감사 로그는 상세 드로어로 분리

#### QC

- 템플릿 관리, 점검 결과, 후속조치 큐를 한 화면에 다 섞지 않는다.
- 최소 3개의 운영 하위 영역으로 분리한다.
  - Template
  - Inspection Results
  - Follow-up Queue

#### Settings

- 매장 운영 모드, 프린터, PIN 등 실제 변경 작업만 보여 준다.
- 진단/테스트/변경 이력은 세부 영역으로 내린다.

#### Deliberry Settlement

- 미정산, 정산대기, 분쟁, 완료 상태를 큐/컬럼으로 먼저 보여 준다.
- 합계 수치는 위쪽에 두되, 실제로 무엇을 처리해야 하는지부터 보이게 한다.

#### E-Invoice

- 실패, 지연, 대기, 수동 개입 필요 건을 상태별 큐로 분리한다.
- dispatch/polling 설정 상태는 경고 배너로 고정한다.
- 결제 화면에서 이 탭으로 drill-down되는 구조를 강화한다.

## 3.5 출결·급여 및 직원 운영

### 목표 플로우

1. 근무 현황 확인
2. 마감 또는 정산 단계 수행
3. 급여 프리뷰 검토
4. 저장/잠금/내보내기

### 제안 구조

- Shift Review 스타일을 참고해 단계형 마감 경험을 만든다.
  - 매출 요약
  - 팁 확인
  - 현금 정산
  - Clock out 또는 close 단계
- 진행 완료 단계는 체크나 색상으로 구분한다.
- 스케줄링 기능이 확장된다면 캘린더 기반 편집 UI를 고려한다.
- 출결 기록 조회와 급여 프리뷰는 연결하되, 조회와 확정 단계를 분리한다.

### 기대 효과

- 출결/급여 기능이 단순 리포트가 아니라 실제 운영 절차로 읽히게 된다.
- 역할 외 정보 노출을 줄이면 매장 직원이 숫자 해석보다 작업 완료에 집중할 수 있다.

## 3.6 딜리버리·테이크아웃 운영

### 목표

- 주문 준비 상태와 배달 상태를 운영 큐 중심으로 보여 준다.

### 제안 구조

- 딜리버리/테이크아웃은 상태 기반 칸반 보드로 정리한다.
  - Ready
  - En Route
  - Delivered
  - Exception
- 카드에는 아래 정보를 고정 노출한다.
  - 주문번호
  - 고객명
  - 주문 유형
  - 예상 시간
  - 현재 상태
- 매장용 Order Ready Board가 필요하면 별도 고객 노출 모드로 분리한다.
  - In Progress
  - Ready

### 기대 효과

- 매장 직원, 포장 담당자, 배달 운영자가 같은 주문을 서로 다른 목적에 맞게 볼 수 있다.
- 주문 상태를 색상과 컬럼으로 분리하면 음성 확인이나 수기 확인 의존도가 줄어든다.

## 3.7 로열티·기프트카드·모바일 주문

### 로열티/기프트카드

- 결제 화면 메인 영역에는 기본 결제 수단만 노출한다.
- `Rewards` 또는 `기타 결제`에서 회원 조회를 연다.
- 전화번호, 이메일, QR 스캔 기반 조회를 지원한다.
- 한 주문에 적용 가능한 계정 수는 제한해 로직을 단순하게 유지한다.

### 모바일 주문

- QR 주문은 별도 손님용 플로우로 분리한다.
  1. 이름/전화번호 입력
  2. 메뉴 탐색
  3. 장바구니
  4. Start a tab 또는 Pay my tab
- 그룹 주문은 기본 플로우에 섞지 말고 향후 옵션으로 둔다.

### 기대 효과

- 직원용 POS와 고객용 주문 경험이 서로 방해하지 않는다.

## 3.8 멀티매장·본사 관리

### 목표

- 본사 화면은 "모든 정보를 다 보여 주는 곳"이 아니라 "우선순위 경고와 드릴다운 시작점"이 되어야 한다.

### 제안 구조

- `photo-ops`와 `super-admin` 모두 큐 중심 대시보드로 재정렬한다.
- 첫 화면은 아래 같은 우선순위 블록으로 구성한다.
  - 매출 이상
  - 재고 경고
  - 출결 이슈
  - QC 미해결
  - E-Invoice 실패
- 이후 매장별 요약 리스트를 두고, 선택 시 특정 매장으로 drill-down한다.
- 멀티매장 접근 모델은 유지하되, 선택된 매장이 항상 명확히 표시되도록 한다.

### 기대 효과

- 본사는 보고서 소비보다 운영 개입 우선순위를 빠르게 판단할 수 있다.
- 매장별 drill-down 구조가 고정되면 멀티매장 학습 비용도 낮아진다.

## 3.9 전자세금계산서 및 지역 특화 기능

### 원칙

- 이 기능은 숨겨야 할 복잡성이 아니라, GLOBOS POS의 핵심 차별점이다.
- 다만 결제 메인 플로우와 예외 처리 플로우는 반드시 분리해야 한다.

### 제안 구조

- 결제 직후:
  - 증빙 사진 저장
  - E-Invoice 백그라운드 큐 등록
  - 발행 접수 상태만 간단히 표시
- 별도 예외 큐:
  - Failed
  - Pending Polling
  - Manual Review
  - Resolved
- 운영자는 이 큐에서만 수동 개입한다.
- 캐셔는 정상 처리 여부만 알고 넘어가도록 한다.

### 기대 효과

- 지역 특화 기능의 중요성은 유지하면서, 현장 직원의 인지 부하는 줄일 수 있다.

## 4. 추가 검증 원칙

재설계는 문서만으로 끝내지 말고, 실제 현장 사용성 검증 단계까지 포함해야 한다.

### 역할별 검증 질문

- 웨이터는 3초 안에 다음에 눌러야 할 버튼을 찾을 수 있는가
- 캐셔는 결제 완료까지 불필요한 판단 없이 진행할 수 있는가
- 주방은 오래된 주문과 긴급 주문을 즉시 구분할 수 있는가
- 관리자는 가장 자주 하는 작업을 검색 없이 시작할 수 있는가
- 본사 운영자는 어느 매장에 먼저 개입해야 하는지 첫 화면에서 알 수 있는가

### 사용성 테스트 제안

- 역할별 시나리오 기반 태스크 테스트를 진행한다.
- 측정 항목은 최소 아래를 포함한다.
  - 첫 액션까지 걸린 시간
  - 작업 완료 시간
  - 오작동 또는 되돌아가기 횟수
  - 교육 없이 수행 가능한지 여부
  - 주관적 스트레스 수준

### 롤아웃 원칙

- 모든 화면을 한 번에 바꾸지 않는다.
- `waiter`, `cashier`, `kitchen`처럼 매장 핵심 흐름부터 단계적으로 바꾼다.
- 이후 `inventory`, `QC`, `E-Invoice`, `HQ dashboard`로 확장한다.

## 5. 우선순위별 실행 제안

### 1단계: 메인 운영 화면 단순화

- 웨이터 화면을 `테이블 선택 -> 주문 편집 -> 전송` 구조로 정리
- 캐셔 화면을 `주문 선택 -> 결제 버튼 -> 후속 단계` 구조로 정리
- 주방 화면 상태 컬럼을 명확히 분리

### 2단계: 예외 처리 워크스페이스 분리

- E-Invoice 현황을 캐셔 메인 흐름에서 더 분리
- QC follow-up, 딜리버리 정산, 재고 승인/입고 예외를 별도 큐로 정리

### 3단계: 관리자 탭 재구성

- 각 탭을 리스트/검색/상세 패턴으로 통일
- 설정과 이력을 본 작업 화면에서 분리

### 4단계: HQ 운영 대시보드 강화

- Photo Ops와 Super Admin을 우선순위 큐 기반으로 재구성
- 다매장 경고와 드릴다운 흐름 표준화

## 6. 결론

GLOBOS POS는 Toast와 같은 기본 운영 골격을 공유하지만, 실제 제품 성격은 더 넓은 운영 플랫폼에 가깝다. 따라서 Toast의 장점을 그대로 복제하는 것이 아니라, Toast의 단순한 운영 흐름 원칙을 가져오고 GLOBOS POS의 지역 특화/본사 운영 기능은 별도 큐와 워크스페이스로 분리하는 것이 가장 적절하다.

핵심은 다음 한 문장으로 요약된다.

> GLOBOS POS는 Toast처럼 단순한 역할 중심 흐름을 가져가되, 베트남 로컬 운영과 본사 운영에서 생기는 복잡성은 메인 플로우가 아니라 별도 예외 큐와 전용 워크스페이스로 흡수해야 한다.
